## B. RESERVA TEMPORAL DE STOCK CON TTL

Problema: stock se descontaba solo al aprobar pago o admin completar orden.
Entre medio, otros clientes podían pedir el mismo stock (race condition).

Solución implementada:
- Nuevas columnas BD:
  - `products.reserved_stock REAL DEFAULT 0`
  - `orders.stock_reserved INTEGER DEFAULT 0`
  - `orders.expires_at DATETIME`
- Nuevo módulo `backend/lib/stockReservation.js`:
  - `reserveForOrder(orderId)`: suma qty a `reserved_stock`, pone expires_at +15min
  - `releaseReservation(orderId)`: libera reserva (idempotente)
  - `expireStalePending()`: cancela pedidos pending vencidos
  - `startScheduler(60_000)`: corre cada 60s
- Stock disponible al público = `products.stock - products.reserved_stock`
  (se expone en `GET /api/products`).
- TTL configurable: `STOCK_RESERVATION_TTL_MIN` (default 15).
- Liberación en webhook Bold approved, admin completed, admin cancelled, expiración.

## G. DESCUENTO 10% PRIMERA COMPRA (SOLO REGISTRADOS)

### Backend
- Nuevas columnas BD:
  - `orders.discount_amount REAL DEFAULT 0`
  - `orders.discount_reason TEXT`
- Nuevo módulo: `backend/lib/discount.js`
  - `computeFirstPurchaseDiscount(userId, subtotal)` — regla:
    usuario autenticado + sin órdenes con status processing|completed
    ni con discount_amount>0 + flag settings activo + percent>0.
  - Autoritativo: nunca confía en cuerpo del request.
- `POST /api/orders` aplica descuento antes de calcular total.
  Total = (subtotal - discount) + delivery_cost.
- Nuevo endpoint: `GET /api/orders/discount-preview?subtotal=N`
  Devuelve {authenticated, amount, percent, reason} para el frontend.
- Nuevas settings:
  - `first_purchase_discount_enabled` (default '1')
  - `first_purchase_discount_percent` (default '10')
  Registradas en `backend/schemas/settings.js` WRITABLE_KEYS y PUBLIC_KEYS.
